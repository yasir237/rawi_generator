# RAWI Generator — راوي

مولّد قوالب سوشال ميديا لتعليم التركية. تدخل بيانات (عنوان، كلمات، خلفية...)
ويطلع لك تصميم جاهز بمقاس Instagram (1080×1080) قابل للتصدير كصورة PNG.

**الفلسفة الأساسية:** ما نصمم صورة، نبني **مكوّنات (Layers)** قابلة لإعادة
الاستخدام. أي منشور جديد = تغيير بيانات فقط، بدون أي لمسة تصميم يدوية.

---

## المكدس التقني (Tech Stack)

- **React 19** + **Vite**
- **TypeScript**
- **Tailwind CSS v4** (بدون `tailwind.config.js` — الإعداد عبر `@theme` في CSS)
- **html-to-image** لتصدير القوالب كـ PNG بمقاس حقيقي

> **⚠️ ملاحظة مهمة عن Tailwind v4:** الكلاسات لازم تكون **نصوص ثابتة**
> بالكود (static strings) عشان Tailwind يقدر يكتشفها وقت البناء (scan).
> `` className={`text-${align}`} `` **ما تشتغل** — Tailwind ما يعرف وقت
> البناء إنها بتصير `text-center` مثلاً، فالكلاس ما يتولّد أبدًا. أي كلاس
> ديناميكي لازم يمر بخريطة ثابتة (`Record<string, string>`) فيها كل
> القيم المحتملة مكتوبة صراحة، أو يُستبدل بـ inline `style` مباشرة (مثل
> `textAlign`). راجع مثال فعلي بقسم `Footer.tsx` تحت.

## 1. نظام التصميم (`design/tokens.ts`)

كل قيمة بصرية (لون، خط، مقاس) تُستورد من هنا فقط. **ممنوع أي مكون أو قالب
يكتب Hex يدوي.**

```ts
colors.bgTop / colors.bgBottom   // تدرج خلفية القوالب الداكنة
colors.primary                    // الأحمر الأساسي (البادج، أيقونة النصيحة بـ ThreeWords، بطاقة "لا تقل" بـ DontSay، ترجمة WordOfDay ⚠️ استثناء، بادج SentenceOfDay)
colors.purple / colors.purpleTop  // البنفسجي الأساسي (يُستخدم بكثافة بـ FeaturedWord: البادج، الأيقونات، TipBanner؛ وبطاقة "قل بدلاً منها" بـ DontSay؛ وبادج/سبيكر/هايلايت WordOfDay؛ والنقطة الفاصلة بقائمة SentenceOfDay)
colors.surface / colors.surfaceMuted // أسطح بيضاء (خلفية الكرت الأبيض بـ FeaturedWord، ورقة TornNote، دوائر الأيقونات، خلفية دائرة السبيكر بـ WordOfDay، خلفية كانفاس SentenceOfDay)
colors.ink / colors.inkMuted      // نصوص داكنة + نصوص ثانوية (تُستخدم داخل الكرت الأبيض، وكل نصوص DontSay لأن خلفيته فاتحة؛ الكلمة التركية ونص السؤال التفاعلي بـ WordOfDay؛ كل نصوص SentenceOfDay)
colors.accent                     // برتقالي — كلمات مميزة بـ FeaturedSentence (نطق، CTA)
colors.noteInk                    // كتابة داكنة (نيفي بنفسجي) فوق ورقة TornNote — أغمق من ink العادي
colors.success                    // أخضر (#22C55E)، دائرة علامة الصح بكبسولة "قل بدلاً منها" بـ DontSay
colors.whiteMuted90                // أبيض بشفافية 90% (rgba(255,255,255,0.9))، نص CtaBox — GramerOfDay
colors.whiteOverlay18               // أبيض بشفافية 18% (rgba(255,255,255,0.18))، خلفية دائرة الأيقونة بـ CtaBox — GramerOfDay
colors.purpleSoft                    // ⭐ جديد — بنفسجي فاتح جداً (#F2EEFC تخميني) — خلفية صندوق القائمة
                                      // + إطار الكانفاس الخارجي بـ SentenceOfDay
wordAccentPalette                 // ألوان تتكرر على كل كلمة (بنفسجي/أحمر بالتناوب) — WordCard فقط
fonts.arabic / fonts.latin        // Alexandria (عربي) / Inter (لاتيني)
layout.canvasSize                 // 1080 — ثابت لكل قالب
layout.padding                    // 64 — المسافة من الحواف، تُستخدم أيضًا لحساب عرض كرت FeaturedWord وعرض صندوق SentenceOfDay
```

> **ملاحظة:** `WordCard` الحالي ما يستخدم `colors.surface` كخلفية —
> خلفيته بالكامل تدرج مبني من `getWordAccent()`. بالمقابل `WordOfDayCard`
> يستخدم `colors.surface` كخلفية بيضاء صلدة كاملة (بدون تدرج).

> **⚠️ استثناء بقالب QuestionAnswer:** الكرت الزجاجي (glass card)
> بمنطقة الجواب يستخدم قيم `rgba(255,255,255,...)` مباشرة بدل tokens —
> هذا يخالف تقنيًا قاعدة "ممنوع Hex/قيم لون يدوية". القرار كان مقصود
> وبطلب صريح (راجع تفاصيله بقسم القالب الرابع تحت)، لكنه **مو نمط
> معتمد للمشروع بشكل عام** — أي قالب مستقبلي يحتاج شفافية بيضاء مشابهة
> يفضّل يضيف tokens جديدة بـ `tokens.ts` (مثلاً `colors.glassBg`,
> `colors.glassBorder`) بدل تكرار كتابة `rgba` يدوي بكل قالب.

| `CtaBox` | شريط CTA بنفسجي صلد بحواف علوية مدوّرة فقط (`rounded-t-3xl`، بدون تدوير سفلي — يعطي إحساس "لسان" ملتصق بأسفل شي) + نص + دائرة أيقونة بخلفية بيضاء شفافة | GramerOfDay |
| `Chip` | كبسولة صغيرة مضغوطة (`inline-flex`، تلتف حول محتواها) بإيموجي/أيقونة + نص قصير، بديل عام عن مستطيل معلومات كامل العرض | WordOfDay |

### تصميم `WordCard` بالتفصيل

البطاقة **بالكامل تدرج واحد موحّد** (مصدر واحد، بدون طبقتين تدرج منفصلتين
تسببان اختلاف لون) — مبني هيك:

```
┌─────────────────────────────────────────────┐
│ [تدرج accent كامل الكرت — نفس الاتجاه للكرت والرقم]│
│  ┌────────┐  ⬤ ← دائرة بيضاء (أيقونة) متراكبة   │
│  │   1    │     على حافة منطقة الرقم/النص      │
│  │(رقم أبيض)│  Teşekkürler   (أبيض)              │
│  └────────┘  شكراً           (أبيض شفاف 85%)     │
└─────────────────────────────────────────────┘
```

- **الخلفية الكاملة للكرت** (`background: gradient`) — تدرج `150deg` من
  `accent` إلى نسخة أغمق منه عبر `color-mix(in srgb, accent 65%, black)`.
- **صندوق النص شفاف بالكامل** (بدون أي خلفية بيضاء) — يبين التدرج من ورا
  النص مباشرة، بدون أي قطع لوني.
- **النصوص بيضاء** (مو داكنة) لأن الخلفية صارت ملوّنة بالكامل — التركية
  أبيض 100%، الترجمة العربية أبيض بشفافية 85%.
- **الدائرة البيضاء** (`ICON_SIZE = 80px`) هي العنصر الأبيض الوحيد في
  البطاقة، تحتوي على `WordIcon` بلون الـ accent.
- **النقاط الزخرفية** يمين البطاقة بيضاء شفافة (`opacity: 0.3`)، مو بلون
  accent (لأنها كانت تختفي فوق خلفية بنفس اللون تقريبًا).

```tsx
<WordCard index={1} turkish="Teşekkürler" arabic="شكراً" icon="heart" />
```

### تصميم `TipBox` بالتفصيل (ThreeWords)

جدًا في تصاميم AI العامة)، التصميم الحالي مبني بنفس **لغة WordCard** بدل
عناصر زخرفية عامة:

- شريط لوني صلد رفيع (`8px`) بلون `colors.purple`، ملاصق بزاوية دائرية
  من جهة واحدة فقط.
- بدون أي دائرة تدرج، بدون `backdrop-filter`/`blur`.
- عنوان صغير ("نصيحة اليوم") بحجم *eyebrow* فوق النص الأساسي — نمط
  تحريري بدل بطاقة تنبيه عامة.

```tsx
<TipBox text="احفظ كلمة واحدة واستخدمها اليوم في محادثتك" />

> **ملاحظة تحديث:** رغم إن هذا القرار (تجنّب Glassmorphism) صار قاعدة
> عامة بالمشروع لفترة (راجع القاعدة الذهبية رقم 6 بالأسفل)، قالب
> `QuestionAnswer` كسرها لاحقًا بقرار صريح ومقصود من صاحب المشروع — راجع
> تفاصيل هذا الاستثناء بقسم القالب الرابع تحت. يعني القاعدة تبقى الافتراض
> العام، لكنها مو مطلقة لو فيه طلب واضح بخلاف ذلك لقالب معيّن.

### تصميم `WordOfDayCard` بالتفصيل

الكرت الأبيض الكبير مبني كطبقات فوق بعضها (`position: relative` +
عناصر `absolute` بداخله):

```
┌───────────────────────────────────────────┐
│ 🏷️(Ribbon فوق الحافة شوي)                    │
│                                              │
│   Günaydın            (عنوان لاتيني ضخم)      │
│   [ gün-ay-dın ] 🔊                           │
│   ┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄┄            │
│   ⬤ المعنى                    ⬤(صورة دائرية) │
│      صباح الخير                 تتكئ على     │
│   ⬤ مثال                        زاوية الكرت   │
│      Günaydın, nasılsın?        من تحت-يمين   │
│      صباح الخير، كيف حالك؟                    │
└───────────────────────────────────────────┘
```

- **عرض الكرت ثابت** = `layout.canvasSize - layout.padding * 2` (952px) —
  نفس منطق الـ padding المستخدم ببقية القوالب، **ما يُكتب رقم يدوي منفصل**.
- **`RibbonTag`** موضعه `top: -24` (يطلع فوق حافة الكرت العلوية بمقدار
  24px عمدًا) و `left: 40` — يعطي إحساس "علامة مثبّتة" على الكرت.
- **عمود النص محدود العرض عمدًا** (`maxWidth: width - 70 - 260`) عشان ما
  يتداخل نص "مثال"/"المعنى" مع `CircleImage` المتكئة يمين-تحت.
- **`CircleImage`** موضعها `right: -60, bottom: -70` — تتكئ على زاوية
  الكرت من برّه (bleed مقصود، بنفس فلسفة `teaImage` بـ ThreeWords)، مع
  حلقة SVG منقطة أكبر منها بـ 40px حواليها.
- **`InfoRow`** تتكرر مرتين (المعنى / مثال)، كل صف = دائرة أيقونة ملونة
  + تسمية + محتوى حر (`children`) يسمح بتنسيق مختلف لكل صف (نص واحد
  للمعنى، سطرين تركي+عربي للمثال).

```tsx
<WordOfDayCard
  turkish="Günaydın"
  phonetic="gün-ay-dın"
  meaning="صباح الخير"
  exampleTurkish="Günaydın, nasılsın?"
  exampleArabic="صباح الخير، كيف حالك؟"
  image="/assets/istanbul-tea.png"
/>
```

> **⚠️ ملاحظة دقة موثّقة:** الـ props أعلاه (`exampleTurkish`/`exampleArabic`،
> بدون `width`) هي المطابقة فعليًا لاستدعاء `WordOfDayCard` داخل
> `FeaturedWord.tsx` الحقيقي (تأكدنا منها بعد رفع الكود الفعلي). **بالمقابل
> الوصف التفصيلي فوق** (بنية `RibbonTag`/`InfoRow`/`CircleImage`، منطق
> `width` كـ prop محسوب بالقالب، حقول `exampleHighlight`/`exampleRest`)
> **موروث من نسخة توثيق أقدم ولسا ما تحقق من مطابقته لمحتوى
> `WordOfDayCard.tsx` الفعلي** (الملف نفسه لسا ما اترفع بهالمحادثة). خذ
> هالوصف كخلفية معمارية عامة مفيدة، لكن لا تعتمد عليه حرفيًا (أسماء props
> داخلية، قيم بكسل دقيقة) لحد ما يُتحقق من الملف الحقيقي — نفس منهج
> "الحالة غير مؤكدة" المتبع بمشروعك لملفات ثانية زي `UiIcon.tsx`.

### تصميم `UiIcon` بالتفصيل — مكوّن أيقونات عام موحّد

**قاعدة مهمة:** أي أيقونة زخرفية عامة (مو مرتبطة بكلمة معينة زي
`WordIcon`) تُضاف هنا، **ما تُنشأ كمكوّن منفصل جديد**. سبق وجرّبنا إنشاء
مكوّن `DecorIcon` منفصل وقت بناء `FeaturedSentence`، واكتشفنا إنه يخالف
القاعدة الذهبية (تكرار نظام أيقونات بدل توسيع الموجود) — فتم دمجه داخل
`UiIcon` وحذف `DecorIcon` نهائيًا. **نفس القصة تكررت مع `BrushBadge`
أثناء بناء `WordOfDay`** — راجع "تطوّر البادج" بقسم القالب السابع.

> **ملاحظة بخصوص `DontSay`:** أيقونتي × (خطأ) و✓ (صح) بهذا القالب
> **ما اتضافت لـ `UiIcon`** — اتبنت كـ SVG صغير محلي **داخل** مكوّن
> `StatusCapsule` نفسه، لأنها مرتبطة عضويًا بشكل الكبسولة (حجمها، سماكة
> الخط، لونها الأبيض الثابت) ومو أيقونة عامة يحتمل تُستخدم بسياقات
> تصميم ثانية بمقاسات/ألوان مختلفة (بعكس `heart`/`quote`/`coffee` مثلاً
> اللي فعلاً استُخدمت بأكثر من مكان). لو احتجنا مستقبلاً نفس شكل ×/✓
> بقالب ثاني بسياق مختلف عن الكبسولة، حينها الأنسب نرفعهم لـ `UiIcon`
> كمفاتيح عامة (`"close"`/`"check"`) بدل تكرارهم محليًا.

```ts
export type UiIconKey =
  | "star" | "translate" | "chat" | "speaker" | "calendar" | "lightbulb" // الأصلية
  | "heart" | "sun-cloud" | "quote"                                      // أُضيفت لأجل FeaturedSentence — quote يُستخدم أيضًا بـ SentenceOfDay
  | "coffee";                                                            // ⚠️ أُضيفت لأجل QuestionAnswer —
                                                                          // موثّقة هنا بس لسا ما اندمجت
                                                                          // فعليًا داخل UiIcon.tsx، راجع
                                                                          // "حالة الدمج" بقسم القالب الرابع
```

- كل أيقونة مسار (`path`) واحد داخل `viewBox="0 0 20 20"` — أي أيقونة
  جديدة مستقبلية لازم تُصمَّم بنفس المقياس عشان تنسجم بصريًا مع الباقي.
- `filledIcons` (حاليًا: `star`, `heart`) هي الأيقونات المصمتة (fill
  بدون stroke منطقي) — البقية outline (stroke بدون fill). `coffee`
  outline زي الأغلبية. `speaker` (المستخدمة بـ WordOfDay لزر النطق)
  outline أيضًا، من الأيقونات الأصلية.
- شكل `heart` مصمم بمنحنيين Bezier متناظرين تمامًا حول المحور العمودي
  (كل إحداثية يسار لها مرآة يمين مطابقة رياضيًا: `x` و `20-x`) — أي
  تعديل مستقبلي على القلب لازم يحافظ على هذا التناظر وإلا يطلع القلب مايل.

```tsx
<UiIcon icon="heart" color={colors.purple} size={40} />
<UiIcon icon="sun-cloud" color={colors.purpleTop} size={80} />
<UiIcon icon="quote" color={colors.purpleTop} size={56} />
<UiIcon icon="coffee" color={colors.purpleTop} size={40} />
<UiIcon icon="speaker" color={colors.purple} size={22} />
<UiIcon icon="quote" color={colors.purpleSoft} size={110} /> {/* SentenceOfDay — زخرفة، الثانية بـ transform: rotate(180deg) */}
```

### تصميم `StatusCapsule` بالتفصيل (DontSay)

كبسولة (Pill) نصية كبيرة بلون صلد حسب `tone`، وعلى حافتها اليمنى دائرة
أيقونة صغيرة متكئة (bleed بسيط، نفس فلسفة `RibbonTag`/`CircleImage`):

```
┌──────────────────────────┐
│                            │╭──╮
│       لا تقل               │ × │  ← دائرة حمراء (tone="wrong")
│                            │╰──╯     أو خضراء (tone="correct")
└──────────────────────────┘
```

- **`tone="wrong"`**: خلفية الكبسولة `colors.primary` (أحمر)، ودائرة
  الأيقونة نفس اللون (أحمر) بعلامة × بيضاء.
- **`tone="correct"`**: خلفية الكبسولة `colors.purple` (بنفسجي)، لكن
  دائرة الأيقونة **`colors.success`** (أخضر) — مو نفس لون الكبسولة، لأن
  الأخضر هو لغة "الصح" العالمية بغض النظر عن لون الكبسولة نفسها.
- الدائرة عندها حدّ أبيض سميك (`border: 5px solid colors.surface`) عشان
  تنفصل بصريًا عن لون الكبسولة تحتها وتبان "طافية" فوقها.
- النص داخل الكبسولة عربي، `fontFamily: fonts.arabic`، `font-black`.
- أيقونتي × و✓ SVG محلي بسيط (`stroke`، بدون `fill`) — راجع الملاحظة
  بقسم `UiIcon` فوق لسبب عدم رفعهم لمكوّن الأيقونات العام.

```tsx
<StatusCapsule label="لا تقل" tone="wrong" />
<StatusCapsule label="قل بدلاً منها" tone="correct" />
```

## 4. القوالب (`templates/`)

### ThreeWords

القالب الأول — يعرض 3 (أو أي عدد) كلمات تركية بترتيب:

```
Logo (أعلى يسار، قريب من الحافة)
  → Title + Badge (عمود واحد، منزاح شوي يسار عن المنتصف)
  → قائمة WordCard (كلمة تلو الثانية، بتدرج موحّد لكل بطاقة)
  → TipBox (أسفل يسار — يظهر دائمًا إذا تم تمرير `tip`)
  → صورة توضيحية (`teaImage`, أسفل يمين — ملتصقة بالحافة تمامًا، بدون
    أي padding/margin، خارج حاوية `layout.padding`)
  → Footer (أسفل المنتصف)
```

```tsx
<ThreeWords
  title="كلمات تركية"
  badge="تستخدمها كل يوم"
  background="aurora"
  footerHandle="@rawi.turkish"
  tip="احفظ كلمة واحدة واستخدمها اليوم في محادثتك"
  teaImage="/assets/tea-simit.png"
  words={[
    { turkish: "Teşekkürler", arabic: "شكراً", icon: "heart" },
    { turkish: "Rica ederim", arabic: "العفو", icon: "handshake" },
    { turkish: "İyi günler", arabic: "يوم سعيد", icon: "sun" },
  ]}
/>
```

> **مهم — طبقات الـ z-index بـ `ThreeWords`:**
> الصورة التوضيحية (`teaImage`) على `z-index: 20`، والمحتوى الرئيسي
> (اللوجو، البطاقات، الفوتر) على `z-index: 10` داخل حاوية منفصلة.
> بمعنى: الصورة تعوم **فوق** المحتوى بصريًا إذا تداخلوا مكانيًا. لو تبي
> العكس (المحتوى فوق الصورة دائمًا)، بدّل الرقمين.

### ThreeWordsExample

معاينة جاهزة تقرأ من `data/three-words-example.json`، فيها أزرار لتبديل
نمط الخلفية حيًا، وزر "تصدير PNG".

**قاعدة مهمة:** مرّر بيانات الـ JSON للقالب بصيغة **spread كامل**
(`{...data}`) بدل استخراج كل حقل يدويًا — لأنه أي حقل جديد تضيفه بالـ
JSON مستقبلاً ينوصل تلقائيًا للقالب بدون ما تحتاج تعدّل ملف الـ Example
كل مرة وتنسى حقل:

```tsx
// ✅ صح — يوصل كل حقل تلقائيًا
<ThreeWords {...data} background={background} />

// ❌ خطأ شائع — لازم تتذكر تضيف كل حقل جديد يدويًا، وسهل تنسى وحد
<ThreeWords
  title={data.title}
  badge={data.badge}
  words={data.words}
  footerHandle={data.footerHandle}
  tip={data.tip}
  // teaImage نسيته هنا مثلاً → الصورة ما تطلع أبدًا رغم وجودها بالـ JSON
/>
```

### FeaturedWord

القالب الثاني — يعرض كلمة تركية واحدة بشرح مفصّل (لفظ، معنى، مثال) داخل
كرت أبيض، بترتيب:

```
Logo (أعلى يسار، خارج حاوية المحتوى الرئيسية)
  → عنوان "كلمة مميزة" + IconPill "Öne Çıkan Kelime" (عمود واحد، متمركز)
  → WordOfDayCard (الكرت الأبيض الكبير — يحتوي Ribbon + عنوان الكلمة +
    لفظها + صفي InfoRow (معنى/مثال) + CircleImage متكئة على الزاوية)
  → TipBanner (صندوق نصيحة بنفسجي صلد، عرض ثابت متمركز)
  → Footer (أسفل المنتصف، `position: absolute` عند `bottom: 32`)
```

```tsx
<FeaturedWord
  turkish="Günaydın"
  phonetic="gün-ay-dın"
  meaning="صباح الخير"
  exampleTurkish="Günaydın, nasılsın?"
  exampleArabic="صباح الخير، كيف حالك؟"
  tip="استخدمها في الصباح عند التحدث مع الأصدقاء أو شراء شيء."
  image="/assets/istanbul-tea.png"
  footerHandle="@rawi.turkish"
  background="orbit"
/>
```

> **✅ تحديث دقة (بعد رفع الكود الفعلي):** الحقول الحقيقية بـ
> `FeaturedWordData` هي `exampleTurkish`/`exampleArabic` (نص كامل +
> ترجمة، بدون تقسيم هايلايت داخل الجملة التركية) — **مو**
> `exampleHighlight`/`exampleRest` زي ما كان موثّق سابقًا بنسخة قديمة من
> هالملف. يعني القالب حاليًا **ما يلوّن** جزء من الجملة التركية داخل
> `WordOfDayCard` (بعكس الوصف القديم تحت اللي كان يفترض تلوين جزئي —
> ذاك الوصف غير مطابق للتطبيق الحالي، احتفظنا فيه فقط كخلفية تاريخية
> لفكرة كانت مخطط لها ولسا ما اتنفذت، أو نُسخة أقدم تغيّرت لاحقًا).

> **ملاحظة تصميم (⚠️ غير مؤكدة — راجع الملاحظة فوق) — لماذا عرض
> `WordOfDayCard` يُمرَّر كـ prop (`width`) بدل ما يُحسب جوّا المكوّن؟**
> عشان `WordOfDayCard` يضل مكوّن عام قابل لإعادة الاستخدام بأي عرض مستقبلاً
> (مثلاً لو صار فيه قالب ثاني بعرض كرت مختلف)، والقالب (`FeaturedWord.tsx`)
> هو المسؤول عن حساب العرض حسب `layout.padding` — نفس مبدأ "التصميم
> يُحسب بالقالب، المكوّن ينفّذ بس". **تنبيه:** استدعاء `WordOfDayCard`
> بالكود الفعلي المرفوع لـ `FeaturedWord.tsx` **ما يمرّر `width` إطلاقًا**
> — يعني إما `WordOfDayCard` يحسب عرضه داخليًا فعلاً (والوصف هذا قديم/غير
> دقيق)، أو فيه قيمة افتراضية داخلية تُستخدم. غير مؤكد لحد ما يُراجع
> ملف `WordOfDayCard.tsx` الحقيقي.

### FeaturedSentence

القالب الثالث — منشور "إعلاني" هدفه إيقاف المستخدم عن التمرير خلال ثانية:
جملة تركية واحدة كبيرة جدًا (البطل الحقيقي للتصميم)، بدون أي معلومات
إضافية (بدون تفكيك، بدون قواعد، بدون أمثلة — هذي تروح لمنشورات ثانية).
الفلسفة: **One Post = One Idea**. بترتيب:

```
Logo (أعلى يسار)
  → PaintedBadge "جملة مميزة" (صورة ثابتة فيها النجمة + التسطير)
  → الجملة التركية (أكبر عنصر، بدون Card ولا إطار، بيضاء + كلمة بنفسجية)
    + UiIcon "quote" (زخرفة أعلى-يسار) و"sun-cloud" (زخرفة أعلى-يمين)
    + خط فرشاة تحتها
  → PhoneticCapsule (Pill شفاف بحدود + زر سبيكر دائري متدرج + نطق عربي)
  → TornNote (ورقة ممزقة حقيقية فيها الترجمة + شريط لاصق)
  → CTA (نص + كلمة مميزة برتقالي + أيقونة Bookmark بدائرة بنفسجية)
  → Footer (أسفل المنتصف)
```

```tsx
<FeaturedSentence
  turkishSentence="Bugün hava çok güzel."
  turkishHighlight="hava"
  phonetic="بوغون هافا تشوك غوزال"
  phoneticHighlight="هافا"
  translation="الجو اليوم جميل جداً"
  translationHighlightWord="جميل"
  cta="احفظها الآن و استخدمها في حياتك اليومية!"
  ctaHighlight="الآن"
  footerHandle="@rawi.turkish"
  background="swoosh"
/>
```

#### ميكانيزم الهايلايت الموحّد (جملة كاملة + كلمة هايلايت)

**قاعدة مهمة لأي قالب جديد بعد هذا:** لتلوين كلمة وحدة داخل جملة، مرّر
**الجملة كاملة كنص طبيعي متواصل** + **الكلمة المطلوب تمييزها لحالها**،
والمكوّن نفسه يسوي `text.split(highlightWord)` ويلوّن الجزء الوسط:

```tsx
const [before, after] = turkishSentence.split(turkishHighlight);
// ...
<span>{before}<span style={{ color: colors.purpleTop }}>{turkishHighlight}</span>{after}</span>
```

**ليش هذا أفضل من تقسيم الجملة يدويًا لحقول منفصلة (`beforeHighlight`/
`afterHighlight`)؟**
- أول نسخة من `FeaturedSentence` كانت تستخدم 3 حقول منفصلة
  (`turkishBeforeHighlight`/`turkishHighlight`/`turkishAfterHighlight`)
  + `<br />` **مثبت بالكود** بين الأجزاء — هذا سبّب مشكلة: كسر السطر كان
  مفروض يدويًا بغض النظر عن طول النص الفعلي، فيطلع القطع "مقطّع" وغير
  طبيعي.
- بعد التوحيد: الجملة نص واحد متواصل، والالتفاف (wrap) للسطر التالي صار
  طبيعي بالكامل حسب `maxWidth` بالحاوية — بدون أي منطق تصميم مخفي بالـ JSON.
- **أي قالب جديد يحتاج تلوين كلمة داخل جملة يستخدم هذا النمط**، مو نمط
  `exampleHighlight`/`exampleRest` القديم بـ FeaturedWord (يضل موجود بس
  لأجل التوافق العكسي فقط).
- **تطبيق إضافي بقالب QuestionAnswer:** نفس الميكانيزم استُخدم مرتين
  بقالب واحد — مرة على السؤال (`question`/`questionHighlight`) ومرة على
  الجملة التركية (`turkishAnswer`/`turkishHighlight`)، يعني ما فيه مانع
  من استخدامه أكثر من مرة بنفس القالب لحقول مختلفة.
- **تطبيق إضافي بقالب WordOfDay:** يُستخدم مرة ثالثة على السؤال التفاعلي
  (`engagementQuestion`/`engagementHighlight`) — راجع قسم القالب السابع.

#### تصميم `PaintedBadge` بالتفصيل

بادج "جملة مميزة" — **صورة PNG ثابتة**، مو SVG مرسوم بالكود، ومو نص +
مكوّن ديناميكي. القرار هذا مقصود:

- النص ثابت دائمًا ("جملة مميزة") — هذا عنوان القالب نفسه، مو بيانات
  متغيّرة حسب كل منشور (بعكس `turkishSentence` مثلاً)، فما نخسر أي مرونة
  بتثبيته كصورة.
- محاولة تقليد خربشة فرشاة AI بدقة عبر SVG path يدوي ما توصل لنفس
  الملمس الحقيقي (تدرج سماكة الحبر، تفاصيل حواف النجمة) مهما حاولنا —
  فالحل الأنسب صورة جاهزة مطابقة تمامًا للمرجع.
- نفس منطق `Logo.tsx` بالضبط: صورة PNG شفافة الخلفية، `import` عادي من
  `src/assets/`.

```tsx
<PaintedBadge height={140} />
```

⚠️ **مهم:** الصورة المصدر (لو جاية من أداة توليد صور AI) قد تحتوي على
خلفية بيضاء صلدة بدل شفافة — تأكد قبل الاستخدام إن قناة alpha فعلاً
شفافة بالزوايا (افحص بـ `img.getpixel()` أو أداة مشابهة)، وإلا الصورة
تطلع بمربع أبيض حولها فوق أي خلفية داكنة.

> **⚠️ لو فكّرت تستخدم نفس أسلوب `PaintedBadge` (صورة PNG) لبادج نص
> جديد بقالب مستقبلي:** راجع أولاً "تطوّر البادج" بقسم القالب السابع —
> جُرِّب هذا الأسلوب لبادج `WordOfDay` (تحت اسم `BrushBadge`، نسخة SVG
> عامة مو PNG) وتم التراجع عنه لصالح `Badge` الموجود. مو كل قالب جديد
> يحتاج بادج مخصص.

#### تصميم `TornNote` بالتفصيل — تقنية 3-Slice بصور حقيقية

الترجمة تظهر داخل "ورقة ممزقة" واقعية — بُنيت من **صور مقصوصة حقيقية**
(مو SVG zigzag يدوي) لأن الملمس الفعلي لتمزق الورق صعب محاكاته بمنحنيات
SVG بنفس مستوى الواقعية. لحل مشكلة "كيف تتمدد الورقة حسب طول النص بدون
ما تتمطط الحواف الممزقة نفسها":

```
┌────┐┌─────────────────────────┐┌────┐
│Left││   Middle (flex-1)        ││Right│
│Cap ││   نسيج متكرر repeat-x    ││ Cap │
│ثابت││   يتمدد حسب طول النص      ││ثابت │
└────┘└─────────────────────────┘└────┘
        [الشريط اللاصق — موضع ثابت فوق]
```

- **`note-left-cap.png` / `note-right-cap.png`**: قطعتان ثابتتا العرض،
  كل وحدة فيها الحافة الممزقة **الحقيقية** (يسار/يمين) من الصورة
  الأصلية — ما تتمطط أبدًا بغض النظر عن طول النص.
- **`note-middle-tile.png`**: شريحة نظيفة من نفس الورقة (مقصوصة من
  منطقة بعيدة عن الشريط اللاصق)، توضع كـ `background-image` بخاصية
  `background-repeat: repeat-x` على `div` بـ `flex: 1` — يتمدد أوتوماتيكي
  حسب عرض النص الفعلي. هذا ينفع فقط لأن حواف الورقة **فوق وتحت** شبه
  مستقيمة (التمزق الحقيقي بس بالحواف الجانبية يسار/يمين) — تأكد من هذا
  الشرط قبل تطبيق نفس التقنية على شكل ورقة ثاني.
- **`note-tape.png`**: صورة الشريط اللاصق منفصلة تمامًا، توضع فوق
  بـ `position: absolute` بموضع ثابت (`left: 50%`) — **ما يتحرك ولا
  يتمدد** مع طول النص.
- **النتيجة:** الجملة **ما تطلع أبدًا خارج إطار الورقة**، لأن العرض
  الكلي = leftCap + rightCap + (عرض المنتصف المطلوب فعليًا حسب طول النص).

```tsx
<TornNote
  text="الجو اليوم جميل جداً"
  highlightWord="جميل"
  minWidth={600} // أقل عرض ممكن — يكبر تلقائيًا لو النص أطول
/>
```

> **كيف تقص صور 3-slice جديدة لشكل ورقة/شريط مختلف مستقبلاً؟**
> 1. تأكد الصورة المصدر عندها قناة alpha شفافة فعليًا (مو خلفية بيضاء
>    صلدة) — افحص بالبكسل قبل أي شي.
>    `img.getbbox()` بمكتبة Pillow يقص حسب المحتوى الفعلي تلقائيًا.
> 2. افحص بصريًا وين التمزق/التفاصيل غير القابلة للتكرار (عادة الحواف
>    الجانبية بشريط أفقي طويل) وقصّها كقطعتين ثابتتين (`leftCap`/`rightCap`).
> 3. قص شريحة نظيفة من منطقة "عادية" بالمنتصف (بعيدة عن أي عنصر ثاني
>    زي شريط لاصق) لتصير قطعة التكرار (`middleTile`).
> 3. أي عنصر إضافي فوق (شريط لاصق، ختم، ريبون...) يُقص لحاله كصورة
>    منفصلة ويوضع بـ `position: absolute` بموضع ثابت.

#### تصميم النطق (`PhoneticCapsule`) والـ CTA

- **كبسولة النطق**: Pill بحدود شفافة رفيعة (`rgba(255,255,255,0.18)`)،
  زر سبيكر دائري بتدرج `purpleTop → purple`، النص محاط بأقواس `[ ]`
  والكلمة المميزة بلون `colors.accent` (برتقالي).
- **CTA**: نص أبيض + كلمة مميزة برتقالي (نفس `colors.accent`) + أيقونة
  Bookmark بدائرة بنفسجية صلدة — نفس ميكانيزم `split()` الموحّد المذكور
  فوق.

#### طباعة النص العربي — دروس مستفادة (`TornNote`)

النص العربي داخل الورقة الممزقة مرّ بتحسينات طباعة محددة بعد ملاحظة إنه
يحس "خشن":

| القيمة | قبل | بعد | السبب |
|---|---|---|---|
| `fontWeight` | 900 (`font-extrabold`) | `700` | الوزن الأثقل بالعربي يلاصق الحروف ببعض بالأحجام الكبيرة، يفقدها رشاقتها |
| `letterSpacing` | (بدون) | `-0.3px` | تقارب خفيف يخلي الكلمة تحس متماسكة كوحدة بصرية |
| `lineHeight` | (بدون تحديد) | `1.4` | مساحة تنفس فوق/تحت النص |
| `fontFeatureSettings` | (بدون) | `"kern" 1, "liga" 1, "calt" 1` | تفعيل الربط والتموضع الدقيق بين الحروف المتجاورة — بعض المتصفحات تعطّلها افتراضيًا |
| `textShadow` | (بدون) | `0 1px 0 rgba(0,0,0,0.06)` | ظل خفيف جدًا يعطي إحساس حبر حقيقي على الورقة بدل نص رقمي مسطّح |

> **قاعدة عامة لأي نص عربي كبير بقالب جديد:** ما يكفي بس اختيار
> `fonts.arabic` والوزن — راجع الجدول فوق كنقطة انطلاق لأي نص عربي
> بحجم كبير (فوق 28px تقريبًا) يحتاج يحس "مصمم" مو "مطبوع افتراضي".

#### تسطير الكلمات المميزة — خط فرشاة مو مستقيم

أي تسطير تحت كلمة مميزة (بالجملة التركية أو بالترجمة العربية) يُرسم
بمنحنى SVG (`path` بمنحنيات بسيطة) **مو** بـ `div` مستطيل مستقيم — نفس
لغة التصميم "المرسوم باليد" المستخدمة بباقي عناصر القالب (خط الفرشاة
تحت الجملة، التسطير المزدوج بالبادج). التسطير يمتد تلقائيًا بعرض الكلمة
عبر `preserveAspectRatio="none"` على الـ `viewBox`:

```tsx
<svg viewBox="0 0 100 12" preserveAspectRatio="none" style={{ width: "calc(100% + 8px)" }}>
  <path d="M2,6 C20,2 35,9 50,5 C65,1 80,8 98,4" stroke={colors.purple} strokeWidth={4} fill="none" strokeLinecap="round" />
</svg>
```

### QuestionAnswer

القالب الرابع — منشور "سؤال وجواب": سؤال شائع عن الحياة اليومية بتركيا
("كيف تطلب X؟") + الجملة التركية المستخدمة فعليًا كجواب + ترجمتها
العربية فقط (**بدون نطق، بدون تفكيك** — بعكس FeaturedWord). الفلسفة: منشور
تطبيقي مباشر (سؤال → جواب جاهز للاستخدام) بترتيب:

```
صف الشعارين (شعار عربي يمين — الإنجليزي يسار لسا معلّق، راجع "الحالة الحالية" تحت)
  → QuoteDivider (خط — علامة اقتباس — خط)
  → السؤال (سطرين، أبيض + كلمة بنفسجية هايلايت، متمركز)
  → NotchDivider (خط بذيل مثلث، فاصل بين السؤال والجواب)
  → كرت زجاجي شفاف (Glassmorphism) يحتوي:
      الجملة التركية (سطرين، أبيض + جزء بنفسجي هايلايت)
      الترجمة العربية (سطر واحد، أبيض)
      UiIcon "coffee" (زخرفة صغيرة أسفل الكرت)
  → رسمة توضيحية SVG (bleed أسفل-يمين، فوق كل شي بـ z-index عالي)
  → Footer (أسفل المنتصف)
```

```tsx
<QuestionAnswer
  question="كيف تطلب قهوة في تركيا؟"
  questionHighlight="قهوة"
  turkishAnswer="Bir kahve lütfen."
  turkishHighlight="lütfen."
  arabicTranslation="قهوة واحدة من فضلك."
  illustration="/assets/coffee-cup.svg"
  footerHandle="@rawi.turkish"
  background="aurora"
/>
```

#### القرارات المهمة أثناء بناء هذا القالب

**1. رفض الصور الحقيقية — SVG بس:**
بعكس باقي القوالب (اللي تستخدم صور PNG حقيقية بـ `teaImage`/`image`)،
اتفقنا صراحة إن هذا القالب **ما يستخدم صور فوتوغرافية أبدًا** — الرسمة
التوضيحية (كوب القهوة بالمرجع البصري) لازم تكون **SVG** (من storyset أو
مصدر مشابه)، تنحفظ بـ `public/assets/coffee-cup.svg` وتُستخدم عبر
`illustration` prop (نفس آلية `teaImage`/`image` تقنيًا — الفرق بس نوع
الملف SVG بدل PNG، والمنطق البرمجي نفسه: مسار من `public/` مو `import`).

⚠️ **حالة حالية:** الملف نفسه (`coffee-cup.svg`) **لسا ما وصل**. الكود
جاهز يستقبله فورًا (`<img src={illustration} />`)، بس بدون الملف الفعلي
بـ `public/assets/` الرسمة ما راح تظهر.

**2. تطوّر تصميم "فقاعة الاقتباس" — من فقاعة صلدة لخط بسيط:**
أول مرجع بصري (الصورة الأولى) كان فيه فقاعة كلام صلدة (rounded square +
ذيل) فوق السؤال، فبنينا `QuoteBubble.tsx` لهذا الغرض. لما تغيّر المرجع
(الصورة الثانية، الأنظف والأبسط)، صار المطلوب خط — علامة اقتباس — خط
بس، فبنينا `QuoteDivider.tsx` الأخف واستبدلناه بمكانه. **`QuoteBubble`
ضل موجود بالمشروع (غير مستخدم حاليًا)** لأنه قد يفيد قالب مستقبلي يحتاج

```tsx
interface FooterProps {
  handle?: string;

}
```

```tsx

⚠️ **درس تقني مهم اتعلمناه أثناء بناء هذا التوسيع (Tailwind v4):**
أول محاولة استخدمت `` className={`text-${align}`} `` لتوليد كلاس المحاذاة
ديناميكيًا — هذا **ما يشتغل** لأن Tailwind v4 (بدون `tailwind.config.js`)
يسوي scan للكود وقت البناء بحثًا عن كلاسات **ثابتة نصيًا**؛ أي
template literal فيه متغيّر (`${align}`) ما يتولّد له CSS فعلي أبدًا،
فالكلاس يطلع بالـ DOM بس بدون أي تأثير بصري. الحل: خريطة ثابتة
(`Record<string, string>`) فيها كل قيم `align` المحتملة مكتوبة صراحة:

```ts
const alignClass: Record<string, string> = {
  center: "text-center",
  left: "text-left",
  right: "text-right",
};
```

**قاعدة عامة لأي كلاس Tailwind ديناميكي مستقبلي بالمشروع:** لازم يمر
بخريطة ثابتة زي فوق، أو يُستبدل بـ inline `style` مباشر (`textAlign`
مثلاً) — **ممنوع** `` `text-${variable}` `` أو أي صيغة مشابهة تعتمد على
تركيب النص وقت التشغيل.

⚠️ **درس تصميم إضافي:** أول نسخة من `Footer` وصلت وفيها ربط غير مقصود
بين `dark` (لون النص) و`align` (محاذاة النص) — يعني لو `dark={false}`،
قيمة `align` كانت تنتجاهل تمامًا ودايمًا يطلع `text-center`. تم تصحيحه:
الحين `dark` مسؤول **بس** عن اللون، و`align` مستقل تمامًا ويشتغل بغض
النظر عن قيمة `dark`. **قاعدة عامة:** أي `prop` جديد يُضاف لمكوّن مشترك
لازم يبقى مستقل المفعول عن الـ `props` الموجودة سلفًا، إلا لو فيه سبب
تصميمي صريح يربطهم ببعض.

#### تحديث `TornNote` — دعم أسطر متعددة (Multi-line)

`TornNote` الأصلي (راجع تفاصيله الكاملة بقسم FeaturedSentence فوق) كان
مصمم لسطر واحد بس — الحاويات `leftCap`/`rightCap` (صور PNG) بارتفاع
ثابت `210px`، والنص `whitespace-nowrap` (يمنع أي كسر سطر). عشان
`GramerOfDay` يحتاج يعرض **قائمة أمثلة** (سطر لكل مثال) داخل **نفس**
شكل الورقة الممزقة (بدل تكرار المكوّن كذا نسخة منفصلة)، اتوسّع المكوّن
بالشكل التالي:

- الـ prop `text` الحين تقبل نص فيه `"\n"` — كل سطر يُرسم لحاله بعمود
  عمودي (`flex-col`) بدل سطر واحد أفقي.
- `leftCap`/`rightCap` صار ارتفاعهم **ديناميكي** (`paperHeight`) بدل
  الرقم الثابت `210` — يُحسب هيك:
  ```ts
  const BASE_HEIGHT = 210; // نفس القيمة الأصلية لسطر واحد
  const EXTRA_PER_LINE = 54; // ⚠️ تقدير تقريبي، يحتاج ضبط بصري بعد أول تصدير فعلي
  const paperHeight = lines.length <= 1
    ? BASE_HEIGHT
    : BASE_HEIGHT + (lines.length - 1) * EXTRA_PER_LINE;
  ```
- **القيد التقني المهم:** بما إن `leftCap`/`rightCap` صور PNG حقيقية
  (مقصوصة من ورقة فعلية بحافة ممزقة)، تكبير `height` يعني **تمديد
  الصورة عموديًا** (scale). هذا مقبول بصريًا لأن التمزّق الحقيقي بس
  بالحواف الجانبية (يمين/يسار) — نفس الشرط المذكور بالتوثيق الأصلي
  لـ `TornNote` (فوق/تحت شبه مستقيمة). النتيجة: الورقة تكبر تناسبيًا
  (عرض وارتفاع مع بعض شوي) كل ما زاد عدد الأسطر — **مو خلل**، هذا سلوك
  متوقّع بحكم طبيعة الصور، ولازم يُختبر بصريًا لعدد أمثلة مختلف (2، 3،
  4 أمثلة) للتأكد إن الشكل يضل مقبول.
- `highlightWord` الحين يشتغل **بس على أول سطر يحتوي عليه** — القائمة
  ما فيها آلية "كل مثال يهايلايت كلمة مختلفة" حاليًا (كل الأمثلة تشترك
  بنفس منطق الهايلايت العام). لو احتجت هايلايت مختلف لكل مثال مستقبلاً،
  لازم تُغيَّر بنية الـ `props` بالكامل (مثلاً `lines: { text: string;
  highlight?: string }[]` بدل `text: string` واحد).
- **`FeaturedSentence` غير متأثر إطلاقًا** — تمريره نص سطر واحد (بدون
  `"\n"`) يعطي نفس النتيجة القديمة بالضبط (`paperHeight = BASE_HEIGHT
  = 210`)، فهذا التوسيع **متوافق للخلف بالكامل**.

```tsx
const exampleItems = examples.map((e) => `${e.turkish} → ${e.arabic}`).join("\n");
// ...
<TornNote text={exampleItems} />
```

#### تصميم `CtaBox` بالتفصيل

شريط بنفسجي صلد (`colors.purple`) بحواف علوية مدوّرة فقط

كرت كامل مستقل. يحتوي نص RTL + دائرة أيقونة بخلفية بيضاء شفافة خفيفة
(`colors.whiteOverlay18`) فيها أيقونة من `UiIcon`.

```tsx

```

استخدم `colors.whiteMuted90`/`colors.whiteOverlay18` بدل قيم `rgba`
يدوية — بنفس منطق الاستثناء المسجّل بـ QuestionAnswer (Glassmorphism)،
هنا القرار كان الالتزام بالقاعدة الذهبية رقم 3 عبر إضافة tokens جديدة
بدل تكرار `rgba` يدوي، **مو** كسر القاعدة كاستثناء.

> **⚠️ ملاحظة استخدام:** `CtaBox` جُرِّب لاحقًا كخيار لعرض السؤال
> التفاعلي بقالب `WordOfDay` وتم التراجع عنه لصالح فقرة نص بسيطة —
> راجع "السؤال التفاعلي" بقسم القالب السابع تحت. `CtaBox` يبقى مستخدم
> فعليًا فقط بـ `GramerOfDay`.

#### ⚠️ ملاحظة مهمة — تموضع `CtaBox` (قرار مقصود، غير محلول تقنيًا بالكامل)

`CtaBox` مبني بـ `position: absolute; bottom: 0` **بدون** أب صريح فيه
`position: relative` مباشرة فوقه بـ `GramerOfDay.tsx` — الحاوية اللي
يقعد فيها مباشرة (`flex-1 flex flex-col items-center justify-center
gap-7`) ما عندها `relative`، فـ`absolute` يتموضع نسبة لأقرب أب فيه
`position` فعليًا، وهو نفس الحاوية الخارجية اللي فيها `Footer` تقريبًا.
تقنيًا هذا يعني `CtaBox` و`Footer` **يتشاركون نفس منطقة التموضع
التقريبية** (تراكب محتمل)، وبما إن `CtaBox` عنصر `absolute`، هو خارج
تدفق الـ `flex` بالكامل (مكانه بالكود بعد `TornNote` **ما يضمن** ترتيبه
البصري الفعلي — هو دايمًا "يطير" لأسفل الحاوية الأقرب اللي فيها
`position`).

**القرار:** صاحب المشروع شاف النتيجة البصرية الفعلية بعد التصدير
وقرر يبقيها كما هي لأنها عاجبته — **بدون** تعديل هيكلي (يعني بدون نقل
`CtaBox` لمكان `relative` صريح، وبدون تحويله لتدفق طبيعي). هذا **قرار
تصميمي مقصود يخص هذا القالب تحديدًا فقط**، مو نمط رسمي معتمد للمشروع —
أي قالب مستقبلي يحتاج عنصر "ملتصق بالأسفل" لازم يستخدم إما:
1. تدفق طبيعي عادي (بدون `absolute`) لو المطلوب "بعد آخر عنصر بالمحتوى"، أو
2. `bleed` صريح خارج حاوية الـ `padding` بالكامل (نفس منطق `teaImage`/
   `CircleImage`) مع أب `relative` واضح على مستوى الكانفاس نفسه، لو
   المطلوب "ملتصق فعليًا بحافة الكانفاس السفلية".

**قبل أي تعديل مستقبلي على هذا الجزء بالذات**، ارجع لهذي الملاحظة أول
— القرار الحالي مقصود ومو خطأ نُسي تصحيحه.

### WordOfDay (القالب السابع)

القالب السابع — النسخة الجديدة من "كلمة اليوم": كلمة تركية واحدة كبطل
التصميم، بترتيب أبسط بكثير من `FeaturedWord` (بدون كرت أبيض، بدون قسم
مثال منفصل). الفلسفة أقرب لـ `FeaturedSentence` ("One Post = One Idea")
بس مطبّقة على كلمة مفردة بدل جملة، مع سؤال تفاعلي إضافي أسفل الكرت
لتحفيز التعليقات.

```
Logo (أعلى يسار، نسخة dark لأن الخلفية فاتحة)
  → Badge "كلمة اليوم" (بنفسجي، عبر مكوّن Badge الموجود)
  → الكلمة التركية (أكبر عنصر بالتصميم، لاتيني ضخم)
  → النطق (رمادي) + زر سبيكر دائري صغير
  → الترجمة العربية (⚠️ أحمر — راجع "استثناء لون الترجمة" تحت)
  → Chip الاستخدام (إيموجي + نص قصير، بديل عن مستطيل كامل العرض)
  → سؤال تفاعلي (تحفيز تعليقات، بالكلمة نفسها مميّزة بنفسجي)
  → Footer (أسفل المنتصف، نسخة dark)
```

```tsx
<WordOfDay
  turkish="Merhaba"
  phonetic="mer-ha-ba"
  translation="مرحباً"
  usageEmoji="👋"
  usageText="تستخدم عند اللقاء"
  engagementQuestion="💜 هل تعرف كلمة أجمل من Merhaba؟ اكتبها في التعليقات."
  footerHandle="@rawi.turkish"
/>
```

#### ⚠️ ملاحظة تسمية مهمة — إعادة استخدام اسم `WordOfDay`

هذا الاسم كان مُستخدم سابقًا لأول نسخة من قالب الكلمة، وتم تحريره بالكامل
عمدًا (راجع Roadmap تحت القالب الثاني — "إعادة تسمية" لـ `FeaturedWord`)
تحديدًا عشان يصير فاضي بالضبط لهذا القالب: تصميم جديد كليًا بمعنى
مختلف. الاسم **مو تكرار** — هو التطبيق الفعلي للخطة اللي كانت موثّقة
مسبقًا. **`WordOfDayCard`** (المكوّن المشترك بـ `components/` المستخدم
داخل `FeaturedWord`) يبقى بدون تغيير وبدون علاقة بهذا القالب الجديد —
اسمين متشابهين، أصل مختلف كليًا.

#### القرارات المهمة أثناء بناء هذا القالب

**1. الخلفية — فاتحة، بدون صور، مبنية محليًا CSS/SVG:**
مثل `DontSay`، هذا القالب **ما يستخدم** `components/Background.tsx`
إطلاقًا (الفلسفة "هادئة" مو "إعلانية داكنة"). لكن بعكس `DontSay` (خلفية
PNG كاملة)، الطلب هنا كان صريح: **بدون صور نهائيًا**. فالخلفية مبنية من
ثلاث دوال محلية داخل `WordOfDay.tsx` نفسه:
- `SoftTexture` — نقاط دقيقة متكررة (`<pattern>`) بشفافية شبه معدومة
  (`opacity: 0.035`) على كامل الكانفاس.
- `CornerSwirl` — 3 حلقات متحدة المركز بنفسجية شفافة، أعلى-يمين (bleed
  خارج حدود الكانفاس بـ `top: -60, right: -60`).
- `CornerDots` — شبكة نقاط 5×5 بنفسجية خفيفة، أسفل-يسار.

⚠️ **الدوال الثلاث محلية غير مُصدَّرة حاليًا** (مو مكوّنات عامة بـ
`components/`) لأنه أول استخدام واحد بس. لو قالب مستقبلي يحتاج نفس
أسلوب "خلفية فاتحة + زخارف أركان بنفسجية"، الأنسب ترفعهم لمكوّنات عامة
(`components/CornerSwirl.tsx`, `components/CornerDots.tsx`) بدل تكرار
الكود — نفس منطق "وسّع الموجود" المتبع بباقي المشروع.

**2. تطوّر البادج — من محاولة `BrushBadge` جديدة إلى `Badge` الموجود:**
أول محاولة كانت مكوّن جديد `BrushBadge` (كبسولة SVG بحافة غير منتظمة
تحاكي خط ماركر، عام يقبل أي نص — نفس روح `PaintedBadge` لكن SVG مو PNG).
صاحب المشروع **حذف هذا المكوّن نهائيًا** واستخدم `Badge` الموجود أصلاً
بالمشروع (نفس المكوّن المستخدم بـ `ThreeWords`، بلون `color`/`fontSize`/
`padding` قابلين للتحكم) بدل بنائه من جديد — قرار يتماشى مع القاعدة
الذهبية رقم 4 ("استخدم مكونات موجودة قدر الإمكان بدل تكرار الكود")،
حتى لو كان يعني التراجع عن مكوّن جديد اتبنى فعلاً. **`BrushBadge` ما
عاد جزء من هذا القالب — احذف ملفه لو لسا موجود بالمشروع.**

```tsx
<Badge children="كلمة اليوم" color={colors.purple} fontSize={40} padding="8px 45px" />
```

**3. ⚠️ استثناء — لون الترجمة أحمر (`colors.primary`) مو بنفسجي:**
الطلب الأصلي كان صراحة: "الترجمة بلون بنفسجي RAWI، مو أحمر، حتى تبقى
الهوية ثابتة". النسخة النهائية تستخدم `colors.primary` (الأحمر) فعليًا،
بحجم كبير (`fontSize: 84`). هذا **قرار مقصود** — صاحب المشروع راجع
النتيجة البصرية بعد التطبيق وفضّل الأحمر، رغم مخالفته للتوجيه الأصلي
المكتوب. **هذا استثناء خاص بهذا القالب فقط، مو قاعدة عامة جديدة** — أي
قالب مستقبلي يحتاج لون ترجمة يرجع لاستخدام `colors.purple` (الافتراض
القياسي بباقي القوالب) إلا لو صار فيه قرار صريح مشابه لهذا تحديدًا.

**4. مكوّن جديد عام: `Chip`:**
بديل عام لأي "مستطيل معلومات كامل العرض" — كبسولة صغيرة مضغوطة
(`inline-flex`، تلتف حول محتواها بس) بإيموجي/أيقونة + نص قصير. مبني
كمكوّن عام (`components/Chip.tsx`) بنفس فلسفة `StatusCapsule`/`IconPill`
— أي قالب مستقبلي يحتاج "ملاحظة سريعة صغيرة" يقدر يعيد استخدامه بدل
بناء شي جديد.

**5. السؤال التفاعلي (`engagementQuestion`) — نص تحفيز تعليقات، مولّد تلقائيًا:**
حقل جديد يظهر أسفل الكرت مباشرة قبل الفوتر، بفقرة نص بسيطة (`<p>`)
بخلفية شفافة (`colors.inkMuted`). المنطق:
- لو ما مُرِّر بالـ JSON، يُبنى تلقائيًا من `turkish`:
  `` `💜 هل تعرف كلمة أجمل من ${turkish}؟ اكتبها في التعليقات.` ``
- لو مُرِّر نص مخصص بالـ JSON، يُستخدم كما هو.
- لو مُرِّر `""` (فاضي) صراحة، يختفي السطر كليًا لمنشور معيّن.
- الكلمة نفسها (`engagementHighlight`, افتراضي = `turkish`) تُميَّز
  بنفسجي داخل الجملة عبر **نفس ميكانيزم الهايلايت الموحّد**
  (`text.split(highlight)`) المستخدم بباقي القوالب — مو تقسيم يدوي
  لحقول منفصلة، ومو حقل بيانات إضافي منفصل بالـ JSON لتخزين الجزأين.
- ⚠️ **تجربة `CtaBox` ثم التراجع عنها:** جُرِّب استخدام مكوّن `CtaBox`
  (المبني أصلاً لـ `GramerOfDay`) لعرض هذا السطر كشريط بنفسجي صلد، لكن
  تم التراجع لصالح فقرة نص بسيطة، لأن `CtaBox` مصمم أصلاً ليكون "ملتصق
  بأسفل الكانفاس" (حواف علوية مدوّرة فقط) وما كان مناسب لتدفق طبيعي
  بمنتصف عمود المحتوى. **`WordOfDay` لا يستخدم `CtaBox` بالنسخة
  النهائية** — راجع ملاحظة الاستخدام بقسم `CtaBox` فوق.

⚠️ **قياسات تقريبية غير مؤكدة:** `fontSize: 24` و `maxWidth: 720`
لفقرة السؤال التفاعلي تقدير أولي بالعين، زي أغلب القياسات الأولى بباقي
القوالب — راجعها بصريًا بعد التصدير، خصوصًا مع كلمات تركية طويلة تخلي
السؤال الافتراضي المولّد تلقائيًا يطول ويقرب من الفوتر.

"مرونة القائمة" تحت). بعكس باقي القوالب، هذا القالب **الكانفاس نفسه**
كرت أبيض بحواف مدوّرة وإطار بنفسجي فاتح خفيف — بدون خلفية داكنة ولا
`Background.tsx` ولا حتى صورة كاملة (بعكس DontSay)، وبدون CSS/SVG محلي
معقّد (بعكس WordOfDay) — أبسط قالب من ناحية الخلفية لحد الآن.

```
كانفاس أبيض بحواف مدوّرة كبيرة + إطار بنفسجي فاتح رفيع
  Logo (أعلى يسار، النمط القياسي bleed)
  → Badge "عبارة اليوم" (أحمر، متمركز)
  → زخرفة quote بنفسجية فاتحة شفافة (أعلى-يمين)
  → الجملة التركية (كبيرة، لاتيني Bold) + الترجمة العربية تحتها (نفس اللون كحلي)
  → زخرفة quote ثانية (أسفل-يسار، نفس الأيقونة مقلوبة 180°)
  → صندوق بنفسجي فاتح صلد: sectionTitle بالمنتصف + قائمة items (• تركي يسار / عربي يمين)
  → Footer
```

```tsx
<SentenceOfDay
  turkish="Nasılsın?"
  arabic="كيف حالك؟"
  sectionTitle="الردود الشائعة"
  items={[
    { turkish: "İyiyim, teşekkürler.", arabic: "بخير، شكراً." },
    { turkish: "Fena değil.", arabic: "لست بخير." },
  ]}
  footerHandle="@rawi.turkish"
/>
```

#### القرارات المهمة أثناء بناء هذا القالب

**1. لا مكوّنات جديدة — كل شي معتمد على الموجود:**
بعكس أغلب القوالب السابقة (كل وحدة جابت مكوّن عام جديد على الأقل)،
`SentenceOfDay` **ما احتاج ولا مكوّن جديد واحد**:
- الشعار → `Logo variant="dark"` بنفس نمط الـ bleed القياسي.
- البادج → `Badge` الموجود مباشرة (نفس المستخدم بـ ThreeWords/WordOfDay).
- علامتي الاقتباس (أعلى-يمين وأسفل-يسار) → نفس `UiIcon icon="quote"`
  المستخدمة بـ FeaturedSentence، الثانية بس بـ `transform:
  rotate(180deg)` — بدون أي تعديل على `UiIcon.tsx` نفسه.
- صندوق القائمة → `flex` عادي (سطر لكل عنصر، عمودين تركي/عربي)، **بدون**
  `TornNote` — لأن التصميم المرجعي هنا صندوق مستطيل صلد بسيط، مو ورقة
  ممزقة، فما فيه داعي لتعقيد 3-slice غير مبرر بصريًا هنا.

**2. `colors.purpleSoft` — توكن جديد بدل تكرار قيمة يدوية:**
اللون البنفسجي الفاتح جدًا (خلفية صندوق القائمة + إطار الكانفاس الخارجي)
مو موجود مسبقًا بـ `tokens.ts` (`purple`/`purpleTop` غامقين أكثر بكثير).
أُضيف كتوكن جديد بدل كتابة Hex يدوي مباشر بالقالب — نفس المنطق المتبع
سابقًا مع `colors.success` (DontSay) و`colors.whiteMuted90`/
`whiteOverlay18` (GramerOfDay).

⚠️ **القيمة الحالية (`#F2EEFC`) تخمين بالعين من الصورة المرجعية فقط**
— لازم تُقارن بصريًا مع اللون الفعلي وتُضبط قبل الاعتماد النهائي.

**3. مرونة القائمة — تسمية عامة `sectionTitle`/`items` بدل `answerTitle`/`answers`:**
رغم إن المرجع البصري يعرض "ردود شائعة" تحديدًا، الحقول اتسمّت بأسماء
عامة **مقصودة** (`sectionTitle` بدل `answerTitle`، `items` بدل
`answers`) عشان نفس القالب يشتغل بدون أي تعديل كود لمحتوى مختلف كليًا
بنفس الشكل — مثلاً:
- "طرق ثانية لقولها" (مرادفات)
- "استخدمها في هذي المواقف"
- "وبعدها اسأل" (أسئلة متابعة طبيعية)

القالب نفسه ما يعرف ولا يهمه معنى المحتوى، هو بس يعرض `sectionTitle` +
قائمة `items` (تركي/عربي) — بالضبط فلسفة "تعبّي بالبيانات بس" المتبعة
بالمشروع. **هذا نمط تسمية يُفضَّل تعميمه على أي قالب مستقبلي مشابه** —
تسمية عامة بدل تسمية مقفولة على أول استخدام محدد للحقل.

**4. الفوتر — غير ظاهر بالمرجع البصري، أُضيف افتراضيًا:**
الصورة المرجعية لا تُظهر أي Footer بالأسفل. أُضيف بالكود بالنمط
الافتراضي (`dark`, `fontSize={13}`) التزامًا بالقاعدة الذهبية رقم 2
(كل قالب لازم `Background → Logo → ... → Footer`). **إذا التصميم
المطلوب فعليًا بدون Footer، هذا قرار يحتاج تأكيد صريح** — حذفه سطر واحد
بالكود لو تأكدت إنه مو مطلوب.

**5. الكانفاس نفسه هو الكرت — بدون خلفية منفصلة تحته:**
بعكس كل القوالب السابقة (خلفية Background.tsx أو صورة أو CSS محلي
**تحت** محتوى بحاوية padding منفصلة)، هنا الكانفاس 1080×1080 نفسه له
`borderRadius: 48` و`border` بنفسجي خفيف مباشرة — يعني الصورة المصدَّرة
النهائية (PNG) نفسها بحواف مدوّرة وإطار، مو مستطيل حاد الزوايا فيه كرت
داخلي. ⚠️ **افتراض غير مؤكد بعد** — لو المقصود فعليًا كرت أبيض عائم جوه
خلفية أوسع (بدل الكانفاس كله يكون الكرت)، الهيكل يحتاج تعديل.

#### حالة الدمج الحالية لـ SentenceOfDay

| الملف | الحالة |
|---|---|
| `design/tokens.ts` | يحتاج إضافة `colors.purpleSoft` يدويًا (قيمة تخمينية، تحتاج تأكيد بصري) |
| `types/index.ts` | يحتاج إضافة `SentenceOfDayItem`, `SentenceOfDayProps`, `SentenceOfDayData` يدويًا |
| `templates/SentenceOfDay.tsx` | ملف جديد كامل |
| `templates/SentenceOfDayExample.tsx` | ✅ ملف جديد كامل — مكتوب ومطابق لنفس فلسفة باقي ملفات الـ Example (`{...data}`، تحقق `if (cardRef.current)`، بدون أزرار تبديل خلفية لأن الخلفية هنا ثابتة بالكامل مو من `backgroundVariants`) |
| `data/sentence-of-day-example.json` | ملف جديد كامل |

⚠️ **نقاط تحتاج تأكيد بصري قبل الاعتماد النهائي:**
1. وجود/عدم وجود Footer (راجع نقطة 4 فوق).
2. دقة `colors.purpleSoft` مقارنة بالمرجع.
3. كل القياسات (`fontSize`, `padding`, مواضع علامتي الاقتباس) تخمين
   أولي بالعين — لازم تصدّر PNG وتقارن.
4. هيكل "الكانفاس = الكرت" (نقطة 5 فوق) — تأكيد إذا هذا فعلاً المقصود.

- `PostData` → مرتبط بـ `ThreeWords` (`type: "three-words"`)

### ⚠️ قاعدة ذهبية: مسارات الصور حسب المصدر

فيه فرق جوهري بين نوعين من الصور بالمشروع، ولازم تختار المسار الصح حسب
النوع وإلا الصورة **ما تظهر أبدًا** حتى لو الملف موجود فعليًا على القرص:

| نوع الصورة | وين تُحفظ | كيف تُستخدم | مثال |
|---|---|---|---|
| **ثابتة بالتصميم نفسه** (لوجو، بادجات، أجزاء مكوّنات) | `src/assets/...` | `import` عادي بالكود | `import logo from "../assets/logos/logo-white.png"`, `import leftCap from "../assets/note/note-left-cap.png"` |
| **متغيّرة حسب كل منشور** (صور جايه من JSON، PNG أو SVG) | `public/assets/...` | مسار نصي مطلق يبدأ بـ `/` | `"teaImage": "/assets/tea-simit.png"`, `"image": "/assets/istanbul-tea.png"`, `"illustration": "/assets/coffee-cup.svg"`, `"background": "/assets/dont-say-bg.png"` |

**السبب التقني:** أي مسار نص عادي (string) جاي من JSON ما يمر على Vite
وقت البناء (`import` هو الوحيد اللي Vite يتعرف عليه ويحوّله)، فلازم يكون
مسار جاهز يفهمه المتصفح مباشرة من جذر الموقع — وهذا بالضبط اللي يوفره
مجلد `public/`، لأن Vite يخدمه مباشرة من `/` بدون أي معالجة. **القاعدة
نفسها تنطبق سواء كان الملف PNG أو SVG** — النوع (JSON-driven مقابل ثابت
بالتصميم) هو المعيار، مو امتداد الملف. **وتنطبق أيضًا حتى لو الصورة
خلفية كاملة للقالب (`background`) مو مجرد عنصر توضيحي صغير** — راجع
مثال `DontSay` فوق. **`WordOfDay` و`SentenceOfDay` استثناء من هذا
القسم كليًا** — بدون أي صورة إطلاقًا (CSS/SVG محلي فقط بـ WordOfDay،
والكانفاس نفسه بدون خلفية منفصلة إطلاقًا بـ SentenceOfDay)، راجع
قسمي القالب السابع والثامن.

**قبل ما تفتح تذكرة مشكلة "الصورة ما تطلع"، تأكد بالترتيب:**
1. الصورة تفتح لحالها بالمتصفح على `http://localhost:5173/assets/xxx.png`
   (أو `.svg`)؟
2. المكوّن يستقبل الـ prop فعليًا (افحص بـ Inspector إذا `<img src=...>`
   موجود بالـ DOM وقيمة `src` مو فاضية/`undefined`)؟
3. الملف اللي يمرر البيانات للقالب (`ThreeWordsExample.tsx` أو
   `FeaturedWordExample.tsx` أو `FeaturedSentenceExample.tsx` أو
   `QuestionAnswerExample.tsx` أو `DontSayExample.tsx`) يمرر الحقل
   فعليًا — أفضل حل: استخدم `{...data}` بدل استخراج يدوي (انظر القسم 4).
4. لو الصورة صورة PNG ثابتة (`src/assets`) جاية من أداة توليد صور AI:
   تأكد قناة alpha فيها شفافة فعليًا بالزوايا (مو خلفية بيضاء صلدة) قبل
   الاستيراد — وإلا تطلع بمربع أبيض حولها فوق أي خلفية داكنة.

> **المرحلة القادمة (Engine):** محرك يقرأ `type` من ملف JSON ويختار القالب
> المناسب تلقائيًا (`three-words` → `ThreeWords`, `featured-word` →
> `FeaturedWord`, `featured-sentence` → `FeaturedSentence`, `question-answer` →
> `QuestionAnswer`, `dont-say` → `DontSay`, `gramer-of-day` → `GramerOfDay`,
> `word-of-day` → `WordOfDay`, `sentence-of-day` → `SentenceOfDay`،
> وقوالب مستقبلية زي `quiz`, `conversation`...)، يملأه بالبيانات، ويصدّر
> PNG. حاليًا كل قالب يُستدعى يدويًا بملف Example خاص فيه.

## 6. تصدير الصورة (`utils/exportImage.ts`)

```ts
import { exportNodeAsPng } from "../utils/exportImage";

exportNodeAsPng(cardRef.current, "featured-word.png");
```

يصدّر بمقاس `1080×1080` حقيقي (`pixelRatio: 1`) لأن القالب مبني أصلًا
بهذا المقاس. لدقة أعلى (طباعة مثلًا)، زوّد `pixelRatio` بالخيارات:

```ts
exportNodeAsPng(node, "file.png", { pixelRatio: 2 });
```

> **تنبيه TypeScript:** `useRef<HTMLDivElement>(null).current` نوعه
> `HTMLDivElement | null` دايمًا، فتأكد من وجود القيمة قبل تمريرها
> لدوال تتوقع `HTMLElement` (مو `null`):
> ```tsx
> if (cardRef.current) {
>   exportNodeAsPng(cardRef.current, "file.png");
> }
> ```

> **تنبيه توافق:** إذا أي مكوّن مستقبلي يستخدم `backdrop-filter`/`blur`
> (تم تجنّبه عمدًا بـ `TipBox` و`TipBanner` الحاليين لهذا السبب بالضبط)،
> تأكد يظهر صح بالـ PNG المصدّر — بعض مكتبات التصدير ما تدعمه بشكل كامل
> بكل المتصفحات. **قالب `QuestionAnswer` كسر هذا التجنّب عمدًا بكرت
> الجواب الزجاجي (راجع تفاصيله بقسم القالب الرابع) — لازم يُختبر
> التصدير الفعلي والتأكد إن الـ blur يطلع صح، وهذا لسا ما تم التحقق منه.**

## 7. اللوجو (`components/Logo.tsx`)

3 نسخ لازم تكون موجودة فعليًا بـ `src/assets/logos/` بنفس الأسماء:

```
logo-original.png   ← الأصلي
logo-white.png       ← للخلفيات الداكنة (شفاف PNG)
logo-dark.png        ← للخلفيات الفاتحة (شفاف PNG) — تُستخدم بـ DontSay, WordOfDay, SentenceOfDay
```

```tsx
<Logo />                     // white افتراضيًا
<Logo variant="dark" />
<Logo variant="original" height={50} />
```

### ⭐ قاعدة التموضع القياسية لأعلى-يسار (كل القوالب)

الشكل المعياري المتبع فعليًا بكل القوالب (راجع `FeaturedSentence.tsx`
كمثال حي) لوضع اللوجو أعلى-يسار **مو** مجرد `<Logo />` مباشرة داخل
حاوية الـ padding العادية — هو ملفوف بـ `div` منفصل بـ `position:
absolute` وهامش سالب متعمّد، عشان اللوجو "يطلع برّه" حاوية الـ padding
شوي (bleed خفيف) بدل ما يلتزم بنفس مسافة الحواف المستخدمة لبقية
المحتوى:

```tsx
<div
  className="absolute flex items-start"
  style={{ marginTop: -24, marginLeft: -20 }}
>
  {/* اللوجو أعلى يسار */}
  <Logo height={130} />
</div>
```

- `position: absolute` — يخرج اللوجو من تدفق الـ flex الرئيسي للقالب
  (عمود Logo→المحتوى→Footer)، فما يأخذ مساحة تدفع باقي العناصر لتحت.
- `marginTop: -24, marginLeft: -20` — هامش سالب متعمّد يقرّب اللوجو أكثر
  من زاوية الكانفاس مقارنة بموضعه الطبيعي لو التزم بـ `layout.padding`
  العادي (64px) — يعطي إحساس اللوجو "لاصق بالركن" أكثر من بقية المحتوى.
  هذي القيم (`-24`/`-20`) قياس بالعين حسب حجم اللوجو نفسه، مو صيغة
  حسابية ثابتة — لو غيّرت `height` اللوجو بشكل كبير، راجع القيمتين مرة
  ثانية بصريًا.
- `<Logo height={130} />` — الحجم القياسي المستخدم فعليًا بالقوالب
  (أكبر من الـ `height={36}` الافتراضي بالمكوّن نفسه)، لأن الافتراضي
  مصمم لسياقات عامة (موقع، هيدر) مو خصيصًا لزاوية القالب.

**أي قالب جديد يحط اللوجو أعلى-يسار لازم يتبع نفس النمط بالضبط** (نفس
بنية الـ `div` الملفوف، نفس فلسفة الهامش السالب) — **مو** يخترع طريقة
توضّع جديدة، حتى لو القيم الدقيقة (`marginTop`/`marginLeft`) تحتاج ضبط
بسيط حسب تركيبة كل قالب بصريًا. **`WordOfDay` و`SentenceOfDay` يتبعان
هذا النمط بالضبط** (`marginTop: -24, marginLeft: -20`, `<Logo
variant="dark" height={130} />`).

> **⚠️ تحديث بخصوص `DontSay`:** القالب الخامس (`DontSay`) استخدم بدل
> هذا نمط أبسط (`<Logo variant="dark" height={64} />` مباشرة داخل تدفق
> الـ flex العادي، بدون `absolute`/هامش سالب) — هذا **انحراف غير مقصود**
> عن القاعدة القياسية فوق، صار لأنه وقت كتابة `DontSay` ما كان عندنا
> بعد مثال `FeaturedSentence.tsx` الفعلي لنقارن عليه. **يُفضّل تعديل
> `DontSay.tsx` ليتبع نفس نمط الـ `div` الملفوف + الهامش السالب** أعلاه
> (مع تعديل `variant` إلى `"dark"` والحجم حسب التصميم المرجعي) عشان
> يتسق بصريًا وبرمجيًا مع باقي القوالب. راجع الكود المصحح بقسم "تحديثات
> مطلوبة" تحت.

> **⚠️ إضافة معلّقة (QuestionAnswer):** قالب `QuestionAnswer` يحتاج
> شعار رابع — نسخة إنجليزية "RAWI TURKISH" بأيقونة ثعلب (تظهر أعلى-يسار
> بجانب الشعار العربي المعتاد أعلى-يمين). هذا الشعار **مو موجود** حاليًا
> بـ `src/assets/logos/`. لما يتوفر الملف، أضِفه بنفس نمط الموجود
> (مثلاً `logo-rawi-turkish-en.png`) واستورده بـ `QuestionAnswer.tsx`
> مكان الـ `div` الفاضي المذكور بقسم القالب الرابع. لو تكرر الاحتياج
> لشعار إنجليزي بقوالب مستقبلية، الأنسب حينها توسيع `Logo.tsx` نفسه
> بـ variant جديد (مثلاً `variant="english"`) بدل التعامل معه كحالة
> خاصة بقالب وحيد — بنفس فلسفة "وسّع الموجود بدل التكرار".

## القاعدة الذهبية — إضافة قالب جديد

أي قالب جديد (Quote card، آية، اختبار، محادثة...) لازم يلتزم بـ:

1. أبعاد ثابتة `1080×1080` ونفس الـ `padding` من `layout.padding` (إلا
   للعناصر المقصود إلصاقها بالحافة أو زاوية عنصر تمامًا، زي `teaImage`
   بـ ThreeWords أو `CircleImage` بـ FeaturedWord أو الرسمة التوضيحية بـ
   QuestionAnswer — تُبنى خارج حاوية الـ padding عمدًا كـ "bleed" مقصود).
2. نفس فلسفة الطبقات: `Background → Logo → عنوان/بادج → Content → Tip/Illustration → Footer`.
   **استثناء `DontSay`/`WordOfDay`/`SentenceOfDay`:** الخلفية هنا مو
   `Background.tsx` — صورة كاملة (JSON-driven) بـ DontSay، أو CSS/SVG
   محلي بدون صور بـ WordOfDay، أو الكانفاس نفسه هو الكرت بدون طبقة
   خلفية منفصلة بـ SentenceOfDay — راجع تفاصيلهم بقسم القالب الخامس،
   السابع، والثامن.
3. الألوان والخطوط تُستورد دايمًا من `design/tokens.ts` — **ممنوع Hex
   يدوي**. الاستثناء الوحيد المقبول رسميًا: تدرجات مبنية ديناميكيًا من
   token موجود عبر `color-mix()` (زي `WordCard`). **استثناء إضافي غير
   رسمي (بطلب صريح فقط):** الكرت الزجاجي بـ `QuestionAnswer` يستخدم
   قيم `rgba` يدوية — هذا مو نمط معتمد للاستخدام التلقائي بقوالب جديدة،
   راجع التفاصيل بقسم القالب الرابع.
4. القالب يُبنى من مكونات `components/` الموجودة قدر الإمكان بدل تكرار
   الكود — ولو احتجت مكوّن عام جديد (زي `UiIcon`/`InfoRow`/`StatusCapsule`/
   `Chip`)، خليه عام وقابل لإعادة الاستخدام من أي قالب مستقبلي، مو مقفول
   على قالب وحيد. **مثال فعلي على هالقاعدة:** أيقونات
   `heart`/`sun-cloud`/`quote`/`coffee` اتضافت لـ `UiIcon` الموجود بدل
   إنشاء مكوّن `DecorIcon` منفصل، ومكوّن `StatusCapsule` بُني عام (يقبل
   `tone`) بدل ما يكون مقفول على كبسولتي DontSay تحديدًا، ومكوّن
   `BrushBadge` الجديد بـ WordOfDay **اتحذف بالكامل** لصالح استخدام
   `Badge` الموجود أصلاً، و`SentenceOfDay` بُني بالكامل **بدون ولا مكوّن
   جديد واحد** — راجع قسم "تصميم `UiIcon`"، "تصميم `StatusCapsule`"،
   "تطوّر البادج" (قسم القالب السابع)، ونقطة 1 بقسم القالب الثامن فوق.
5. أي حقل بيانات جديد للقالب يُضاف لنوعه الخاص بـ `types/index.ts` كـ
   اختياري (`?`) أول ما يُنشأ، وتُستخدم `{...data}` بدل تمرير حقول يدويًا.
   **مثال على أسماء حقول عامة بدل مقفولة:** `SentenceOfDay` استخدم
   `sectionTitle`/`items` بدل `answerTitle`/`answers` عشان نفس القالب
   يستوعب محتوى مختلف (مرادفات، مواقف استخدام...) بدون تعديل كود — راجع
   "مرونة القائمة" بقسم القالب الثامن.
6. تجنّب عناصر زخرفية "عامة" (دوائر تدرج + `blur` زجاجي) بدون مبرر مرتبط
   بهوية راوي تحديدًا — استخدم نفس لغة التصميم الموجودة (تدرجات
   `WordCard`، الأشرطة اللونية الصلدة، الدوائر المتقطعة حوالين اللوجو
   أو `CircleImage`). **هذه القاعدة قابلة للكسر بطلب صريح ومقصود لقالب
   معيّن (راجع Glassmorphism بـ QuestionAnswer)، لكنها تبقى الافتراض
   العام لأي قالب جديد ما فيه طلب صريح بخلاف ذلك.**
7. أي عنصر يتكئ على حافة/زاوية عنصر ثاني (bleed) لازم يُحسب موضعه بالنسبة
   لأبعاد فعلية (عرض/ارتفاع الكرت أو الكانفاس)، مو أرقام عشوائية — قِس
   من التصميم المرجعي أول لو موجود.
8. **لتلوين كلمة وحدة داخل جملة/نص**، استخدم ميكانيزم "جملة كاملة +
   كلمة هايلايت" الموحّد (`text.split(highlightWord)` داخل المكوّن) —
   **مو** تقسيم النص يدويًا لحقول منفصلة (`beforeX`/`afterX`) بالـ JSON.
   راجع قسم "ميكانيزم الهايلايت الموحّد" بقالب FeaturedSentence. يقدر
   يُستخدم أكثر من مرة بنفس القالب لحقول مختلفة (راجع QuestionAnswer،
   وسؤال WordOfDay التفاعلي).
9. لو القالب يحتاج عنصر بصري صعب محاكاته بـ SVG بواقعية كافية (ورق ممزق،
   خامات، ظلال معقّدة)، استخدم **صورة حقيقية مقصوصة** بدل SVG يدوي —
   وإذا العنصر يحتاج يتمدد حسب طول النص، طبّق تقنية **3-slice** (قطعتين
   ثابتتين بالحواف + منتصف متكرر `repeat-x`) بدل تمطيط الصورة كاملة.
   راجع `TornNote` كمثال مرجعي كامل. **استثناء:** قالب `QuestionAnswer`
   تحديدًا يستخدم SVG (مو صور حقيقية) لرسمته التوضيحية بقرار صريح من
   صاحب المشروع — راجع "رفض الصور الحقيقية" بقسم القالب الرابع.
   **ملاحظة:** قائمة `SentenceOfDay` استخدمت صندوق `flex` صلد بسيط
   (**مو** `TornNote`) لأن التصميم المرجعي هنا مستطيل عادي بدون تمزق —
   لا تستخدم `TornNote` تلقائيًا لكل قائمة، بس للحالات اللي فعلاً
   تحتاج ملمس "ورقة ممزقة" بصريًا.
10. بمجرد ما القالب يصير جاهز ومطابق لهذي القواعد، تعبّيه بالبيانات بس —
    بدون أي لمسة تصميم يدوية بعدها.
11. ⭐ **قاعدة اللوجو أعلى-يسار:** استخدم دايمًا نمط الـ `div`
    الملفوف بـ `position: absolute` + هامش سالب (`marginTop`/`marginLeft`)
    الموضّح بقسم "اللوجو" فوق — **مو** `<Logo />` مباشرة داخل تدفق الـ
    flex العادي. هذا يضمن اتساق التموضع (bleed خفيف عن حاوية الـ padding)
    بين كل القوالب. `WordOfDay` و`SentenceOfDay` يتبعانها بشكل صحيح.
12. ⭐ **قاعدة كلاسات Tailwind الديناميكية ممنوعة:** أي كلاس
    Tailwind يعتمد على قيمة متغيّرة وقت التشغيل (زي محاذاة أو لون يتغيّر
    حسب prop) لازم يمر بخريطة ثابتة (`Record<string, string>`) فيها كل
    القيم المحتملة مكتوبة صراحة بالكود، أو يُستبدل بـ inline `style`
    مباشر. **ممنوع** صيغة `` `text-${variable}` `` أو أي template
    literal مشابه — Tailwind v4 (بدون config) ما يكتشفها وقت البناء
    فينتج كلاس بدون أي تأثير بصري فعلي. راجع مثال `Footer.tsx`
    (`alignClass`) بقسم القالب الخامس.
13. ⭐ **قاعدة عنصر "ملتصق بالأسفل" لازم أب `relative` صريح:**
    أي عنصر يستخدم `position: absolute` (مثل `bottom: 0`) لازم يكون
    أقرب أب له فيه `position: relative` **مقصود ومحدد بوضوح** — إذا
    الحاوية المباشرة ما عندها `relative`، العنصر يتموضع نسبة لأبعد أب
    (عادة حاوية القالب الكاملة)، وقد يتراكب مع عناصر أخرى (زي `Footer`)
    بشكل غير متوقع. قبل استخدام `absolute`، حدد بوضوح: (أ) تدفق طبيعي
    عادي بدون `absolute` أصلاً، أو (ب) `bleed` صريح خارج حاوية الـ
    `padding` بالكامل مع أب `relative` واضح على مستوى الكانفاس (نفس
    منطق `teaImage`/`CircleImage`). **استثناء موثّق:** `CtaBox` بقالب
    `GramerOfDay` يستخدم `absolute bottom-0` بدون أب `relative` مباشر
    بقرار مقصود من صاحب المشروع رغم التراكب المحتمل مع `Footer` — راجع
    تفاصيل هذا الاستثناء بقسم القالب السادس قبل اعتماده كنمط بقالب جديد.
    **`WordOfDay` و`SentenceOfDay` يتجنّبان هذا الموقف بالكامل** —
    عناصرهم الإضافية (السؤال التفاعلي، صندوق القائمة) بتدفق طبيعي عادي
    (بدون `absolute`)، بالضبط الخيار (أ) المذكور فوق.
